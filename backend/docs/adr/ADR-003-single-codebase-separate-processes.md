# ADR-003: 단일 코드베이스 / 별도 프로세스 (API vs Worker)

**상태**: Accepted  
**날짜**: 2026-05-05

---

## 컨텍스트

STDF 파싱 워커와 FastAPI 서버는 역할이 다르다:
- **API 서버**: HTTP 요청 처리, 짧은 응답 시간 중요
- **Worker**: CPU/IO 집약적인 파일 파싱, 긴 처리 시간 허용

두 컴포넌트를 어떻게 배치할지 결정해야 한다.

### 검토한 옵션
- **A. 완전 분리**: 별도 리포지토리, 별도 이미지
- **B. 단일 코드베이스 + 별도 프로세스** ← 선택
- **C. 단일 프로세스**: API 내부에서 백그라운드 태스크로 처리

## 결정

**옵션 B**: 단일 코드베이스에서 두 개의 진입점을 분리한다.

```
# API 서버 진입점
uvicorn app.main:app --host 0.0.0.0 --port 8000

# Worker 진입점
python -m app.worker
```

Docker 컨테이너는 동일 이미지를 빌드하되, `docker-compose.yml`의 `command`로 진입점을 달리한다. `app/models/`, `app/services/` 같은 공유 레이어를 양쪽에서 임포트한다.

## 결과

### 긍정적
- SQLAlchemy 모델, Pydantic 스키마, 비즈니스 로직을 두 컴포넌트가 공유 — 중복 없음
- Worker만 독립적으로 수평 확장 가능 (`docker compose up --scale worker=N`)
- API 서버 장애가 Worker에 전파되지 않음 (독립 프로세스, 장애 격리)
- 이미지 빌드 파이프라인이 하나 — CI 단순

### 부정적 / 트레이드오프
- Worker용 의존성(pystdf 등)이 API 이미지에도 포함됨 — 이미지 크기 소폭 증가
- 완전 분리(옵션 A) 대비 컴포넌트별 독립 배포는 불가 (이미지 재빌드 필요)
- 옵션 C(단일 프로세스) 대비 운영 복잡도 소폭 높음 — 그러나 격리·확장성이 명백히 우위

### 보류 / 운영 시점
- Worker 처리량이 API와 독립적으로 성장하면 별도 리포지토리로 분리 가능
- Worker 이미지 최적화가 필요하면 멀티스테이지 Dockerfile로 분기
